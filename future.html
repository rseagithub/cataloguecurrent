RSEA Future-State Workflow

With the migration to Adobe Commerce, Pronto Xi becomes the master system for catalogue, entitlement and uniform data. This walkthrough focuses on the new Catalogue flow, with a brief overview of the other two streams.

Catalogues

SCENE 10  ·  Overview — Entitlements
Entitlements — what each user can purchase, and how much
Where catalogues control visibility, entitlements control spend and frequency. A user is entitled to specific stock codes (or styles, or groups), with quantity limits, spend caps and renewal cycles. Enforcement happens at Sales Order and POS entry in Pronto Xi.

Lifecycle: overnight batch (1am) checks completed cycles, rolls forward to the next period, and retires expired entitlements. Unused balance does not carry over. eCommerce orders that exceed entitlement values land in Pronto with an On Hold status for review.
SCENE 11  ·  Overview — Uniforms
Uniforms — managing custom-branded apparel requests
Some customers want their logo embroidered onto shirts, or a specific colourway not stocked off-the-shelf. The Uniforms module tracks these requests through a defined approval workflow, with options (colour, embroidery, placement) attached to each master record.

SCENE 12  ·  Cross-Team Alignment — Open Questions
Where the Pronto and Adobe specs diverge
Both teams have written internal specs that don't entirely match. This view surfaces the gaps so they can be resolved before build. Items marked HIGH are blockers; MEDIUM are integration-shape questions; LOW are open items the Adobe team has already flagged.
#1
High
Pronto spec says
Pronto owns the category hierarchy

Categories are defined in Pronto via a dedicated Categories drill-down screen on each catalogue, with custom L1/L2/L3 code tables managed there.

vs
Adobe spec says
Adobe owns the category hierarchy

"Categories themselves are still created and managed in the standard Adobe Commerce way." Pronto only links existing categories via catalog_scope_category.

#2
High
Pronto data shape
Catalogue → Categories → Lines → SKUs

Four objects describe the catalogue's structure: zcus-catalogue-header, zcus-catalogue-categories, zcus-catalogue-lines, zcus-catalogue-line-stock.

vs
Adobe data shape
Catalogue scope as permissions tag

Four tables describe access: catalog_scope, catalog_scope_product, catalog_scope_access, catalog_scope_category. Categories and products live in Adobe's native entities.

#3
Medium
Pronto spec says
Fixed 3-level hierarchy

Cat L1 (mandatory) → Cat L2 (mandatory) → Cat L3 (optional). Hard-coded fields in the line-stock table.

vs
Adobe spec says
Unlimited-depth native tree

Uses catalog_category_entity with self-referencing parent_id. No depth limit; Acidgreen example shows mixed depth subtrees.

#4
Medium
Pronto spec says
One debtor + one role per catalogue

Catalogue header has a single zch-accountcode and zch-role. Multiple role assignments require Copy-ing the catalogue.

vs
Adobe spec says
Many-to-many access grants

catalog_scope_access allows multiple (company, role) pairs per catalogue. role_id = NULL means all roles for that company.

#5
Medium
Pronto spec says
Style entries explode to SKUs

Operator can enter a style (e.g. E1310) and tick which variants apply. Result stored in zcus-catalogue-line-stock.

vs
Adobe spec says
Flat SKU list in payload

POST /catalog-scopes/{id}/products accepts {"skus": [...]}. No notion of styles. Pronto must explode before sending — only line-stock crosses the wire.

#6
Medium
Pronto spec says
No category enrichment

No fields for description, image, or SEO metadata on the category records.

vs
Adobe spec says
Adobe-side enrichment workflow

"Category enrichment is the one operation that happens directly inside Adobe Commerce, performed by RSEA admin users." Pronto data + Adobe enrichment combined.

#7
Open
Pronto spec says
Field details TBD

Source field names for catalog_scope_product not addressed in the Pronto custom-objects diagram.

vs
Adobe spec says
"Missing" on the Pronto side

Data mapping table for catalog_scope_product explicitly shows Missing for both pronto_code and product_id source fields. Adobe team has flagged this.

#8
Open
Pronto spec says
Categories drill-down on each catalogue

Implies hierarchy is owned per-catalogue.

vs
Adobe spec says
"Mismatch mapping" flagged

The catalog_category_entity mapping shows Mismatch mapping for both parent_id and name. Open question already flagged by Acidgreen.

To resolve before build: the two HIGH-severity items above are the must-have decisions — category ownership (#1) and the data-model translation (#2). Without alignment on these, the integration layer between Pronto and Adobe Commerce can't be built unambiguously. The remaining items can be sorted as part of detailed design.
